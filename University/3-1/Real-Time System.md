## 1. 실시간 시스템의 기초 (Basic Concepts)
- **Real-Time System**: 작업의 결과물뿐만 아니라 그 결과가 도출되는 **시간(Timing)**이 시스템의 정식성(Correctness)에 영향을 주는 시스템.
- **Hard Real-Time**: 마감 시간(Deadline) 위반 시 시스템에 치명적인 실패(Catastrophic failure)가 발생하는 시스템.
- **Soft Real-Time**: 마감 시간을 어겨도 성능 저하는 있으나 시스템 전체가 실패하지는 않는 시스템.
- **Predictability (예측 가능성)**: 시스템이 정해진 시간 안에 작업을 완료할 수 있음을 미리 보장할 수 있는 성질.
## 2. 태스크 모델 (Task Model)
- **Periodic Task ($T_i$)**: 일정한 주기($p_i$)마다 반복적으로 발생하는 작업.
    - **Phase ($\phi_i$)**: 첫 번째 Job이 릴리즈되는 시점.
    - **Execution Time ($e_i$)**: 작업이 실행되는 데 걸리는 최대 시간.
    - **Relative Deadline ($D_i$)**: 작업 발생 시점부터 완료되어야 하는 시점까지의 시간 간격.
    - **Absolute Deadline ($d_i$)**: 실제 시스템 시간상의 마감 시점 ($d_i = r_i + D_i$).
- **Aperiodic Task**: 발생 시점이 정해져 있지 않으며(비주기적), 마감 시간이 없거나 유연한 태스크.
- **Sporadic Task**: 비주기적으로 발생하지만, 엄격한 마감 시간(Hard Deadline)을 가진 태스크.
## 3. 스케줄링 파라미터 (Scheduling Parameters)
- **Release Time ($r_i$)**: 작업이 실행될 준비가 되어 큐에 들어가는 시점.
- **Utilization ($U$)**: 프로세서 사용률. 단일 태스크 $u_i = e_i / p_i$이며, 전체 이용률 $U = \sum u_i$.
- **Hyperperiod ($H$)**: 모든 주기적 태스크의 주기의 최소공배수($\text{LCM}$)로, 전체 스케줄링 패턴이 반복되는 단위 시간.
- **Laxity (Slack Time)**: 여유 시간. (남은 마감 시간 - 남은 실행 시간). 이 값이 0이면 즉시 실행해야 함.
## 4. 스케줄링 알고리즘 (Scheduling Algorithms)
- **EDF (Earliest Deadline First)**: 마감 시간이 가장 가까운 작업에 가장 높은 우선순위를 부여하는 동적 우선순위 방식.
- **SJF (Shortest Job First)**: 실행 시간이 가장 짧은 작업 우선.
- **SRTF (Shortest Remaining Time First)**: 선점형 SJF. 남은 실행 시간이 가장 짧은 작업 우선.
- **LRT (Latest Release Time)**: 작업을 마감 시간부터 역방향(Backward)으로 배치하여 실행을 최대한 뒤로 미루는 방식.
- **Effective Release Time / Deadline**: 선후 관계(Precedence)가 있을 때 이를 고려하여 조정한 실제 유효 시간.
## 5. 클록 기반 스케줄링 (Clock-Driven Scheduling)
- **Cyclic Schedule**: 정해진 시간표에 따라 반복적으로 실행되는 정적 스케줄링.
- **Frame ($f$)**: 타이머 인터럽트가 발생하는 최소 시간 단위.
- **Frame 제약 조건**:
    1. $f \geq \max(e_i)$: 프레임은 어떤 작업의 실행 시간보다도 커야 함 (작업 분할 방지).
    2. $H \pmod f = 0$: 하이퍼피리어드는 프레임 크기로 나누어떨어져야 함.
    3. $2f - \gcd(p_i, f) \leq D_i$: 작업이 마감 시간 안에 완료됨을 보장하기 위한 조건.

## 📅 9주차: EDF 밀도 분석 & 클럭 기반 스케줄링 (Frame & INF)

### 1. 마감시간 제약($D < p$)에서의 EDF 밀도 검사 (Density Test)
* **개념**: 마감시간이 주기보다 짧은 태스크가 존재할 때, EDF 알고리즘의 스케줄링 가능성을 판단하는 **보수적인 충분조건** 기법입니다.
* **공식**:
  $$\text{Total Density } (\Delta) = \sum_{i=1}^{n} \frac{e_i}{\min(p_i, D_i)} \le 1.0$$
* **특징**: 밀도의 합이 1.0을 초과하면 공식상으로는 실패(Fail)지만, 이는 충분조건일 뿐이므로 **실제 간트 차트를 그렸을 때는 정상적으로 스케줄링이 가능할 수도 있습니다.** (필요충분조건이 아님)

### 2. 클럭 기반(Cyclic) 스케줄링과 프레임 크기($f$) 결정 제약 조건
* **Major Cycle (Hyperperiod)**: 모든 주기들의 최소공배수($LCM(p_1, p_2, \dots, p_n)$).
* **Frame Size ($f$) 결정을 위한 3대 제약 조건**:
  1. **주기 제약**: $f \le p_i$ (프레임 크기는 각 태스크의 주기보다 작거나 같아야 함)
  2. **실행 제약**: $f \ge \max(e_i)$ (프레임은 가장 긴 실행 시간보다 크거나 같아야 잘림 없이 실행 가능)
  3. **마감시간 제약**: $2f - \gcd(p_i, f) \le D_i$ (태스크가 방출된 후 마감시간 전에 온전한 프레임이 최소 하나 이상 포함되어야 함)

### 3. 네트워크 플로우를 이용한 스케줄링 검증 (INF 그리기)
* **목적**: 결정된 프레임 크기 $f$ 내에 각 Job들을 어떻게 배치할지 결정하기 위해 최대 유량(Max-Flow) 네트워크 알고리즘을 사용합니다.
* **그래프 매핑 구조**:
  * **Source $\rightarrow$ Job 노드 ($J_i$)**: 간선 용량(Capacity) = 해당 Job의 실행 시간 $e_i$.
  * **Job 노드 ($J_i$) $\rightarrow$ Frame 노드 ($F_k$)**: Job이 방출 시간($r_i$) 이후부터 마감시간($D_i$) 이전에 실행될 수 있는 모든 프레임으로 간선을 연결합니다. 간선 용량 = $f$.
  * **Frame 노드 ($F_k$) $\rightarrow$ Sink**: 간선 용량 = 프레임 크기 $f$.
* **스케줄링 가능성 판정**: 알고리즘을 돌려 구한 최종 최대 유량(Max-Flow) 값이 **모든 Job의 실행 시간 총합($\sum e_i$)과 완벽히 일치**하면 스케줄링 가능하다고 판단합니다.



## 📅 10주차: 주기적 태스크 스케줄링 알고리즘 (RM, DM, EDF, LST)

### 1. 정적 우선순위 스케줄링 (Static Priority)
* **RM (Rate-Monotonic)**
  * **규칙**: 주기가 짧은($p_i$가 작은) 태스크에게 높은 우선순위 부여 ($D = p$ 일 때 최적).
  * **이용률(Utilization) 상한선 (충분조건)**:
    $$U = \sum_{i=1}^{n} \frac{e_i}{p_i} \le n(2^{1/n} - 1)$$
    * 태스크 수 무한대($n \to \infty$) 시 최악의 이용률 상한선은 **약 69.3% ($\ln 2$)**입니다.
* **DM (Deadline-Monotonic)**
  * **규칙**: 상대 마감시간이 짧은($D_i$가 작은) 태스크에게 높은 우선순위 부여 ($D < p$ 일 때 최적).

### 2. 동적 우선순위 스케줄링 (Dynamic Priority)
* **EDF (Earliest-Deadline-First)**
  * **규칙**: 현재 시점에서 절대 마감시간(Absolute Deadline)이 가장 임박한 태스크에게 CPU 할당.
  * **이용률 상한선 (필요충분조건)**: $D=p$ 일 때 총 이용률 $U \le 1.0$ (100%) 이면 무조건 스케줄링 가능.
* **LST (Least-Slack-Time-First)**
  * **규칙**: 여유 시간(Slack Time)이 가장 적게 남은 태스크에게 CPU 할당.
  * **여유 시간 계산 공식**: $\text{Slack } (s_i) = (D_i - \text{현재시간}) - \text{남은 실행시간}$



## 📅 11주차: TDA (Time-Demand Analysis, 시간 요구량 분석)

* **목적**: 정적 스케줄링(RM, DM)에서 총 이용률이 충분조건 상한선을 초과했을 때, 실제로 마감시간을 지킬 수 있는지 검증하는 **필요충분조건** 분석 기법입니다. 최악의 타이밍(Critical Instant)을 가정합니다.
* **시간 요구량 함수 공식**:
  $$w_i(t) = e_i + \sum_{j \in hp(i)} \left\lceil \frac{t}{p_j} \right\rceil e_j$$
  *(단, $hp(i)$는 나보다 우선순위가 높은 태스크들의 집합)*

### 🏃‍♂️ $t$ 값 반복 대입 알고리즘 수렴 과정
1. **초기 설정**: 우변의 $t$ 자리에 검사할 태스크의 순수 실행 시간인 **$t^{(0)} = e_i$**를 넣고 우변을 계산하여 결과 값 $w_i(t^{(0)})$를 구합니다.
2. **비교 및 재대입**: 계산되어 나온 결과 값이 내가 대입한 값과 다르면, **그 결과 값을 그대로 다음 $t$ 자리에 대입**합니다.
   * $t^{(1)} = w_i(t^{(0)})$
   * $t^{(2)} = w_i(t^{(1)})$
3. **종료 조건**:
   * **성공(수렴)**: 대입한 값과 나온 결과 값이 완벽히 일치하는 순간($w_i(t) = t$), 해당 $t$가 최악 응답 시간이 됩니다. 이 값이 **$t \le D_i$ 이면 스케줄링 가능**합니다.
   * **실패**: 계산 도중 $t$ 값이 검사 태스크의 마감시간 **$D_i$를 초과하는 순간 즉시 중단 및 스케줄링 불가능**으로 판정합니다.



## 📅 12주차: 비주기적 태스크 스케줄링 (Serverless vs Server)

### 1. 비서버(Serverless) 기반 기법
* **Background 스케줄링**
  * 비주기적 태스크에 최하위 우선순위를 부여합니다. 주기적 태스크가 실행되지 않는 완전한 유휴 시간(Idle time)에만 밑바닥에서 실행됩니다. (응답 속도가 가장 느림)
* **Slack-Stealing 스케줄링**
  * 주기적 태스크들이 마감시간을 어기지 않는 선에서 가질 수 있는 여유 시간(Slack)을 실시간으로 계산합니다. 주기적 태스크를 뒤로 밀고 **비주기적 작업을 앞으로 당겨서 즉시 실행**시켜 응답 속도를 극한으로 끌어올립니다.

### 2. 주기적 서버(Server) 기반 기법 (주기 $P_s$, 예산 $E_s$)
* **Polling Server (폴링 서버)**
  * 주기 시작점에 서버가 깨어났을 때 비주기 큐가 비어있으면 **남은 예산을 즉시 0으로 소멸(`Discharge`)**시킵니다. 주기 중간에 비주기 작업이 도달해도 다음 주기까지 기다려야 합니다.
* **Deferrable Server (DS, 대기성 서버)**
  * 주기 시작점에 작업이 없어도 예산을 소멸시키지 않고 주머니에 쥔 채 **다음 주기가 오기 전까지 대기(`Defer`)**합니다. 주기 중간에 비주기 작업이 오면 높은 우선순위로 즉시 선점하여 처리합니다.
* **Sporadic Server (SS, 산발적 서버)**
  * 예산을 **실제 사용하기 시작한 시점($t_A$)**을 기준으로, 다음 주기($t_A + P_s$)가 찾아왔을 때 **정확히 사용한 양만큼만 다시 충전(`Replenishment`)**해 주는 방식입니다. 주기적 태스크에 가하는 오버헤드가 가장 적습니다.



## 📅 13주차: 공유 자원 동기화 프로토콜 (Priority Inversion)

### 1. Priority Inversion (우선순위 역전 현상)
* **현상**: 하위 태스크($T_L$)가 공유 자원(Mutex/Semaphore)을 소유한 상태에서, 자원과 아무 상관 없는 중간 태스크($T_M$)가 깨어나 $T_L$을 선점해 버립니다. 이 때문에 자원을 기다리던 최상위 태스크($T_H$)까지 중간 태스크가 끝날 때까지 억울하게 블로킹되는 현상입니다.

### 2. PIP (Priority Inheritance Protocol, 우선순위 상속)
* **메커니즘**: 상위 태스크가 하위 태스크가 잡고 있는 자원 때문에 블로킹되는 순간, **자원을 쥔 하위 태스크의 우선순위를 상위 태스크의 우선순위로 강제 격상(상속)**시켜 중간 태스크의 선점을 방지합니다. 자원을 반납(`Unlock`)하면 즉시 원래 원래 계급으로 수직 하락합니다.
* **한계**: **교착 상태(Deadlock)** 및 여러 자원을 거치며 블로킹이 누적되는 Chained Blocking을 방지하지 못합니다.
* **중요 선점 규칙**: 하위 태스크가 상속을 받아 가짜 우선순위(예: 3) 상태로 실행 중이더라도, **그 상속된 우선순위보다 원래 우선순위가 더 높은 진짜 상위 태스크(예: 4)가 방출되면 선점당합니다.**

### 3. PCP (Priority Ceiling Protocol, 우선순위 천장)
* **우선순위 천장(Ceiling)**: 각 공유 자원마다 **"이 자원을 획득할 가능성이 있는 모든 태스크 중 가장 높은 우선순위"**를 미리 부여해 둡니다.
* **자원 할당 규칙**: 어떤 태스크가 자원을 요청할 때, **현재 시스템 내에서 다른 태스크들에 의해 잠겨 있는 모든 자원들의 천장 값 중 최댓값보다 내 원래 우선순위가 엄격하게 높아야만** 자원 획득을 허가합니다.
* **장점**: 자원을 꼬아서 획득하는 상황을 원천 차단하므로 **Deadlock(교착 상태)이 절대 발생하지 않으며**, 상위 태스크는 평생 **단 한 번의 블로킹(At most one block)**만 겪는 구조적 안정성을 수학적으로 보장합니다.
