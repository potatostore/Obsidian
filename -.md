	    - ⑤ 환경 관점: 훈련 과정 환경 영향, 활용 과정 사회적 영향, 범위 범위 구축

개발 Process 모델 
1. 폭포수 모델: 순차적 개발 → 코딩/테스트 지연, 자원 소요 예측 어려움. 요구사항 변동 시 대응 불가.
2. 프로토타입 모델: 프로토타입에서 사용자의 피드백 첨가. 폭포수 모델의 단점 보완. 비용/시간 추가 소요.
3. 변형 모델: 정형 명세서 검토를 통해 확인 후 수행.
4. 나선형 모델: 목표 설정 → 위험 분석 → 개발/테스트 → 평가. 다음 단계 반복.
5. V-모델: 작업과 테스트 매칭 → 높은 신뢰성 요구 분야
6. 진화적 생태 주기 모델: 릴리즈를 통해 변경 작업의 명백한 순서 보장. 복잡하고 비쌈.
7. 점진적 모델: 반복적으로 분석/설계/구현/테스트를 통해 최종 결과물 산출. 변경 쉽고 신속 전달.

  Unified Process (UP) : 도입 → 정례 → 구축 → 전환의 반복적이고 점진적인 방식. 문서화가 잘 되어 교육용 용이. 시간/노력 많이 듬.

  Agile Process(XP) : 개인 간의 소통 중시. 반복-점진 모델. 폭포수 모델 해결. XP, Scrum.
- XP (Extreme Programming): 용기, 존중, 의사소통, 피드백, 단순성 원리. 짝 프로그래밍, 테스트 우선 개발(TDD), 지속적 통합.
- 엔지니어링 사이클: (코딩 → 테스트 → 피드백 → 설계) → 배포
- Scrum Process: 반복적 개발의 관리 과정 강조.

Chaos / DevOps 
- Chaos Engineering: 새로운 기능을 도입하기 전 실험을 통해 장애 예측 및 검사
- DevOps: 개발자+운영자 소통, 협업 → 신속 개발. 자동화, 반복, 자기 서비스, 지속적 개선, 협업. (지속적 테스트, 지속적 통합/배포 CI/CD)

프로젝트 관리: 계획 수립 → 자원 획득 → 실행 → 모니터링 
- 일정 관리: 작업 분할 구조(WBS) / 베이스라인 (요구사항, 기능, 제품) / PERT, 간트 차트
- 비용 관리: COCOMO (예측 어려움, 언어별로 다름), COCOMO II
- 조직 관리: 중앙집중형 구조 / 분산형 구조 / 하이브리드 구조
- 프로젝트 자원 도구: 프로젝트 관리 도구, PMO 도구, 엔지니어링 도구

기능 점수: 처리 기능 + 데이터 기능 
- 산출 방식: ① 기존 점수 산정 유형 선정, ② 범위 및 경계 설정, ③ 데이터 기능 산출 및 복잡도 식별, ④ 처리 기능 산출 및 복잡도 식별, ⑤ 예비 기능 점수 산정
- 간이법에 의해 대략적 예측 용도로 사용 가능.

요구사항: 수행해야 할 것, 속성, 기술 
- 기능적 요구사항 + 비기능적 요구사항 (성능, 보안성, 안정성, 일관성, 유효성) + 인터페이스 요구사항
- 추출 방법: [대면: 인터뷰, JAD], [비대면: 문서 분석, 설문지 활용, 관찰, 소셜 네트워크]
- 요구사항 정의서 ← 기능/비기능/인터페이스 + 품질 요구사항, 개발 요구사항 등
