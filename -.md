-수준: 관점/추상화 수준에 따라 달라짐
UML → 객체지향 소프트웨어를 모델링하는 표준 그래픽 언어

    -객체: 상태, 동작, 고유 식별자를 가진 모든 설계

    -클래스 종류:

        -컨트롤 클래스: 계산/알고리즘 모델화

    -② CRC 카드: 팀 협업 시 효율↑, 결합된 객체를 찾을 수 있음. 유지 보수 용이 → 도메인 전문가 필요
    -③ User Stories: 단일 기능/하나의 작업 표현

    -'-▷' (실선 화살표) : 상속 표시
    -'--▷' (점선 화살표) : 구현 표시
요구 모델링: 요구사항 모델링으로의 전환 도구 
-도메인 분석 → 도메인 개념 → 기능적 모델 → 사용 사례(Use Case)
유스케이스 다이어그램: 시스템의 요구 추출, 분석 
-구성요소:
    -액터: 시스템과 상호작용하는 외부 엔티티
    -사용 사례(Use Case): 시스템 제공 기능, 액터에 의해 사용
-작성 방법: ① 액터 찾기, ② 유스케이스 관계 찾기 → 가치/목적 중심, ③ 유스케이스 명세화 → 정상 흐름, 오류 흐름
-유스케이스 관계 찾기:
    -① 포함 관계 (`<<include>>`): 중복 제어 및 포함
    -② 확장 관계 (`<<extend>>`): 일정 조건 하에 확장
소프트웨어 품질 → 눈으로 확인하기 어렵고, 절대적인 평가 X 
-외적 품질: 정밀성 (1-(A/B)=P), 신뢰성(오류 발생 정도), 완전 견고성(정해지지 않은 조건에도 합리적으로 동작), 성능(효율성), 사용의 편리성(사용 용이), 가용성(정상적으로 사용 가능한 정도), 보안성
-내적 품질: 검증 가능성, 유지보수성, 재사용성(기존 SW Component 재사용), 이식성, 생산성, 상호 운용성(협력 수행 능력)
프로세스 품질 vs 품질 요소 
-프로세스 모델 적합성,개발 방법론 적합성,도구 적합성,표준 준수성,프로젝트 데이터 관리 수준
인공지능 소프트웨어 품질 → 전통적인 소프트웨어 개발 분류로는 그대로 적용 불가 
-품질 특성: 투명성/청렴, 다양성/공평성/사회적 웰빙(특정 요소에 편중 X), 보안/안정성, 견고성/신뢰성, 윤리적 측면
-품질 요소 
	-ML 시스템의 구성 관점:
	    -① 모델 관점: 데이터에 대한 훈련이 이루어지는 부분
	    -② 데이터 관점: 모델에 입력되는 실 데이터
	    -③ 시스템 관점: Component 연결 및 형상 관리 (변경)
	    -④ 인프라 관점: 어떻게 구축되는가에 초점
	    -⑤ 환경 관점: 시스템-사용자의 상호작용 표현
	-관점별 품질 요소:
	    -① 모델 관점: 모델 튜닝, 모델 타당성, 훈련 모델 적합성
	    -② 개발 데이터 관점: 미흡성, 정확성, 일관성, 유효성, 다양성, 일관성
	    -③ 시스템 관점: 출력 처리 효율성, 제어 효율성, 범용 제어 효율성
	    -④ 인프라 관점: 인프라 적합성, 훈련 성능 효율성, 실행 성능 효율성
	    -⑤ 환경 관점: 훈련 과정 환경 영향, 활용 과정 사회적 영향, 범위 범위 구축
품질 모델: McCall-FCM, HP-FURPS, ISO 9126, 25010 
Process 없는 개발: Code-and-Fix (설계 X), 소프트웨어 프로세스: 설계+품질 관리 
개발 Process 모델 
1. 폭포수 모델: 순차적 개발 → 코딩/테스트 지연, 자원 소요 예측 어려움. 요구사항 변동 시 대응 불가.
2. 프로토타입 모델: 프로토타입에서 사용자의 피드백 첨가. 폭포수 모델의 단점 보완. 비용/시간 추가 소요.
3. 변형 모델: 정형 명세서 검토를 통해 확인 후 수행.
4. 나선형 모델: 목표 설정 → 위험 분석 → 개발/테스트 → 평가. 다음 단계 반복.
5. V-모델: 작업과 테스트 매칭 → 높은 신뢰성 요구 분야
6. 진화적 생태 주기 모델: 릴리즈를 통해 변경 작업의 명백한 순서 보장. 복잡하고 비쌈.
7. 점진적 모델: 반복적으로 분석/설계/구현/테스트를 통해 최종 결과물 산출. 변경 쉽고 신속 전달.
소프트웨어 개발에 공통적으로 수행되는 프로세스 1.요구사항 분석 2.설계 3.구현 4.테스트 5.유지보수
인도 후 유지보수 3가지 활동: 1.수정 유지보수: 오류 수정, 2.적응 유지보수: 환경 변화 대응, 3.완전 유지보수: 성능 개선 및 기능 추가
시스템 테스트 데이터 작성 시기: V -모델 기준, 요구사항 분석 또는 시스템 설계 단계에서 테스트 계획과 데이터를 생성하는 것이 가장 적절함.
Unified Process (UP) : 도입 → 정례 → 구축 → 전환의 반복적이고 점진적인 방식. 문서화가 잘 되어 교육용 용이. 시간/노력 많이 듬.
Agile Process(XP) : 개인 간의 소통 중시. 반복-점진 모델. 폭포수 모델 해결. XP, Scrum.
-XP (Extreme Programming): 용기, 존중, 의사소통, 피드백, 단순성 원리. 짝 프로그래밍, 테스트 우선 개발(TDD), 지속적 통합.
-엔지니어링 사이클: (코딩 → 테스트 → 피드백 → 설계) → 배포
-Scrum Process: 반복적 개발의 관리 과정 강조.
Chaos / DevOps 
-Chaos Engineering: 새로운 기능을 도입하기 전 실험을 통해 장애 예측 및 검사
-DevOps: 개발자+운영자 소통, 협업 → 신속 개발. 자동화, 반복, 자기 서비스, 지속적 개선, 협업. (지속적 테스트, 지속적 통합/배포 CI/CD)
프로젝트 관리: 계획 수립 → 자원 획득 → 실행 → 모니터링 
-일정 관리: 작업 분할 구조(WBS) / 베이스라인 (요구사항, 기능, 제품) / PERT, 간트 차트
-비용 관리: COCOMO (예측 어려움, 언어별로 다름), COCOMO II
-조직 관리: 중앙집중형 구조 / 분산형 구조 / 하이브리드 구조
-프로젝트 자원 도구: 프로젝트 관리 도구, PMO 도구, 엔지니어링 도구
기능 점수: 처리 기능 + 데이터 기능 
-산출 방식: ① 기존 점수 산정 유형 선정, ② 범위 및 경계 설정, ③ 데이터 기능 산출 및 복잡도 식별, ④ 처리 기능 산출 및 복잡도 식별, ⑤ 예비 기능 점수 산정
-간이법에 의해 대략적 예측 용도로 사용 가능.
요구사항: 수행해야 할 것, 속성, 기술 
-기능적 요구사항 + 비기능적 요구사항 (성능, 보안성, 안정성, 일관성, 유효성) + 인터페이스 요구사항
-추출 방법: [대면: 인터뷰, JAD], [비대면: 문서 분석, 설문지 활용, 관찰, 소셜 네트워크]
-요구사항 정의서 ← 기능/비기능/인터페이스 + 품질 요구사항, 개발 요구사항 등
스크럼 독특한 3가지 활동: 스프린트, 일일 스크럼 회의, 제품 백로그 관리
소프트웨어공학이 추구하는 3가지 목표: 품질 향상, 비용 절감, 일정 준수
유지보수 4가지: 수정(Corrective), 적응(Adaptive), 완전(Perfective), 예방(Preventive)
기준선 (Baseline): 특정 시점에서의 산출물(Artifact)들의 완전한 버전을 의미하며, 형상 관리의 기준이 됨.
예) 에피소드 4 종료 시점의 요구사항4/분석4/설계4/구현4 문서가 확정되어 다음 단계의 기초가 된 상태.
비용 산정 모델: COCOMO(소프트웨어 규모(KDSI/LOC)를 기반으로 통계적 분석을 통해 비용과 인력을 예측하는 모델, 기능 점수(소프트웨어
가 제공하는 기능의 양과 복잡도를 기준으로 규모를 산정하며, 프로젝트 초기 단계에서도 예측이 가능함.)
CPM: 임계 경로 – 프로젝트의 시작부터 끝까지 가장 긴 시간이 걸리는 경로로, 이 경로상의 직업이 지연되면 전체 일정이 지연됨
계산 요소: ES(빠른 시작), EF(빠른 종료), LS(늦은 시작), LF(늦은 종료)를 구하여 여유 시간(Slack)이 0인 경로를 찾음.
EF = ES + 기간, LS = LF – 기간, LS -ES 또는 LF -EF
WBS: 프로젝트 목표를 달성하기 위해 필요한 전체 작업을 계층적으로 분해한 것.
CPM 작성법: 각 직업의 선후 관계를 화살표로 연결, 임계 경로(Critical Path): 시작부터 끝까지 가장 긴 시간이 걸리는 경로, 이 경로에 있는
작업은 절대 지연되어서는 안 됨., 최소 기간 계산: 임계 경로상의 작업 소요 시간 합계.
간트차트 및 자원 배분: 최소 인원 산출: 동시에 진행해야 하는 작업의 최대 개수를 파악.
가로축은 시간, 세로축은 인원/작업으로 하여 겹치지 않게 배치.
요구사항 구분, 기능적 요구사항: 시스템이 무엇을 하는가(예: 비밀번호 입력 등), 비기능적 요구사항: 품질, 성능 등(예: 3초이내 응답 등)
클래스 수출(명사 추출법): 요구사항 명세서에서 명사를 찾아 클래스와 속성으로 선정하고, 동사를 찾아 연산자로 선정함.
1. COCOMO II의 3단계 예측 모델 (예측 시점 기준)프로젝트 진행 단계에 따라 사용하는 데이터와 정확도가 달라집니다.
특징1단계 애플리케이션 합성 (Application Composition) -프로토타입 단계에서 화면이나 출력 개수 등을 기반으로 객체 점수(Object Point)
를 사용하여 예측함
2단계초기 설계 (Early Design) -시스템 구조 설계 단계에서 기능 점수(Function Point)를 사용하여 예측하며, 구체적인 알고리즘 결정 전
3단계구조 설계 후 (Post-Architecture) -개발이 본궤도에 오른 후 코드 라인(LOC)이나 기능 점수, 보정 계수를 모두 사용하여 가장 정확하
게 예측함
2. 기능 점수(FP)의 5가지 산정 요소 -소프트웨어가 제공하는 기능을 사용자 관점에서 데이터와 트랜잭션으로 나눔
내부 논리 파일 (ILF): 시스템 내부에서 사용자가 관리하고 유지하는 데이터 그룹입니다.
외부 인터페이스 파일 (EIF): 다른 시스템이 소유하고 관리하지만, 우리 시스템에서 참조만 하는 데이터 그룹입니다.
외부 입력 (EI): 경계 밖에서 데이터나 제어 정보를 시스템 내부로 입력하는 기능입니다.
외부 출력 (EO): 가공된 데이터를 사용자나 다른 시스템으로 전달하는 기능입니다.
외부 조회 (EQ): 시스템 내부 데이터의 단순한 검색 및 결과를 출력하는 기능입니다.
3. 애자일 선언문(Agile Manifesto) 4대 가치"왼쪽의 것보다 오른쪽의 가치를 더 중요하게 생각한다"는 점이 핵심입니다.

4. XP(Extreme Programming) 핵심 실천사항
반복적 개발 활동을 극대화하는 12가지 중 시험에 잘 나오는 5가지입니다.짝 프로그래밍 (Pair Programming): 두 개발자가 한 컴퓨터에서 동
료 검토를 거치며 함께 코딩함. 테스트 우선 개발 (TDD): 구현 전에 테스트 케이스를 먼저 작성하여 결함을 조기에 발견함. 리팩토링
(Refactoring): 기능은 그대로 두되 코드 구조를 개선하여 가독성과 품질을 높임. 지속적 통합 (Continuous Integration): 수시로 코드를 통합하
고 테스트하여 오류를 즉시 확인. 공동 소유 (Collective Ownership): 누구나 시스템의 어떤 부분도 수정할 수 있는 권한과 책임을 가집니다.
