IP 프로토콜은 데이터 전송을 담당하는 프로토콜이다.

하지만 전송 과정이 올바르게 이루어지기 위해서는 전송 프로토콜 외에도 다양한 제어 프로토콜이 필요하다



가장 먼저 다뤄볼 것은 IP주소와 MAC 주소 사이의 변환을 담당하는 ARP/RAPR 프로토콜이다.



**ARP**

데이터를 올바르게 전달하기 위해서는 네트워크 체계에서 논리적인 주소를 의미하는 IP뿐 아니라

실제 하드웨어의 위치를 의미하는 MAC 주소도 알아야 한다.

호스트가 데이터를 보낼 때 입력하는 MAC주소는 자신의 LAN 카드에 내장되어있기에 읽어서 저장하면 되지만

문제는 수신 호스트의 MAC 주소를 알 방법이 없다는 것이다.

따라서 수신 호스트의 IP 주소를 이용해 MAC 주소를 알아내야 한다.



**-수신 호스트의 MAC 주소 알아내기**

수신 호스트의 MAC 주소를 알아내기 위해서 ARP request라는 특수 패킷을 브로드캐스팅한다.

해당 패킷은 네트워크의 모든 호스트가 수신하지만 관계없는 호스트들은 패킷을 무시하고,

수신 IP와 본인의 IP가 일치하는 수신 호스트가 ARP reply 패킷에 자신의 MAC 주소를 회신한다.



데이터를 전송할 때마다 ARP request/reply를 사용하는 것은 불필요한 트래픽의 증가로 이어지기에

ARP 캐시를 사용한다.



**RAPR**

디스크가 존재하지 않는 시스템이나 윈도우 터미널 등에서는 자신의 LAN 카드 정보를 읽어

MAC 주소를 얻을 수 있다.

네트워크에는 RARP 기능을 전담으로 수행하는 서버가 있는데 해당 서버가

ARP request 패킷을 받아 MAC 주소를 반환해준다.



**ICMP**

인터넷 환경 오류에 관한 처리를 지원한다.

IP 프로토콜은 데이터 전송 과정에서 패킷 폐기 등의 오류가 발생해도 보고하는 기능이 없다.

따라서 이를 지원하기 위한 네트워크 계층 프로토콜이 필요하고 이것이 ICMP이다.



**-ICMP 메시지**

ICMP에 의해 발생하는 메시지는 Type 필드 값에 따라 구분된다.

* STINATION UNREACHABLE

수신 호스트가 존재하지 않거나, 프로토콜이나 포트 번호 등이 없어 수신 호스트에 접근 불가능한 경우

* SOURCE QUENCH

네트워크 자원이 부족해 패킷이 버려지는 경우에 발생한다.

* TIME EXCEEDED

패킷의 TTL 필드 값이 0이 되어 패킷이 버려진 경우에 주로 발생한다.

* ECHO REQUEST, ECHO REPLY

특정 호스트의 인터넷이 활성화되어 있는지 확인할 수 있다.

* TIMESTAMP REQUEST, TIMESTAMP REPLY

두 호스트 간의 네트워크 지연을 계산하는 용도로 사용한다.



**헤더 형식**

Type, Code, Checksum



**-Type**

1바이트 크기로 메시지의 종류를 구분한다.

Type+Code 필드로 메시지를 구분한다.



**-Code**

메시지 내용에 대한 자세한 정보를 제공하는 매개변수 값이다.



**-Checksum**

ICMP 전체 메시지에 대한 체크섬 기능을 지원한다.



ICMP의 주요 업무는 전송 오류 보고이기에 오류가 발생한 패킷에 대한 정보를 제공해야 한다.

이때, 오류 원인을 제공한 IP 패킷의 헤더와 이어지는 8바이트 정보가 오류 보고 메시지를 포함한다.

ECHO,TIMESTAMP/REQEUEST,REPLY는 질의 메시지이기에 오류 발생 IP 패킷같은 정보가 없다.

이때는 Identitier, Sequence Number필드를 사용해서 메시지를 구분하는 기능이 사용된다.



정리하자면 Type,Code, Checksum은 고정이고 이후에 8바이트 오류 정보가 있는데

이때 질의 메시지는 오류 정보가 없기에 식별자, Sequence Number가 들어간다는 것이다.



**ICMP 메시지 전송**

기능적으로는 IP 프로토콜과 같은 네트워크 계층의 역할을 수행한다.

하지만 바로 데이터 링크 계층에 전달되지 않고 IP 패킷에 캡슐화된 후에 전달된다.

ICMP에 IP 헤더를 추가해 캡슐화하고 다시 데이터 링크 계층으로 넘어간다는 것이다.

###### 

**IGMP 프로토콜**

특정 그룹에 속하는 모든 호스트에 메시지를 전송하는 방식을 멀티캐스팅이라고 한다.

이때 라우팅 기법도 달라지게 되는데 여기에 사용하는 알고리즘을 멀티캐스팅 라우팅이라 한다.



**-그룹 관리**

멀티캐스팅 라우팅에서는 다수의 호스트를 논리적으로 묶기 위한 그룹 관리 기능이 필요하다.

주요 기능으로는 그룹 생성, 제거, 참가, 탈퇴가 있다.



목적지 주소가 그룹 주소로 지정된 패킷을 그룹의 모든 호스트에게 전달하기 위해서

라우터가 입력 패킷의 호스트의 수만큼 복사해 전달해야 한다.



**-멀티캐스팅을 구현을 위해 필요한 기능**

1. 다중 수신 호스트를 표시하는 멀티캐스트 그룹 주소 표기 방법을 통일해야 한다.
2. 라우터에서 IP 멀티캐스트 주소와 이 그룹에 속하는 멤버 호스트의 네트워크 주소 사이의 연관성을 처리할 수 있다.

특히 멤버가 동적으로 추가 및 삭제되는 환경에 대응하는 기능이 필요하다.

3\.    멀티캐스트 라우팅 알고리즘은 그룹에 속한 모든 멤버에게 도달하는 가장 짧은 경로를 선택하는 기능을 제공



그룹의 상태를 알기위해 호스트가 라우터에게 자신의 상태를 알리는 방식이 있고

라우터가 주기적으로 호스트에게 문의하는 방식이 있다.



**IGMP 헤더 구조**

**-Type**

크게 3가지 값을 가진다. 0x11(멀티캐스팅 라우터가 전송한 질의 메시지),

0x16(호스트가 전송하는 보고 메시지)

0x17(그룹 탈퇴에 관한 메시지)

**-Max Response Time**

질의 메시지에서만 사용한다. 최대 응답 시간이다.

**-Checksum**

**-Group Address**

질의 메시지는 0으로 채우고, 보고 메시지에는 호스트가 가입을 원하는 그룹 주소를 표기한다.

특정 그룹 관련 질의 메시지에는 해당 그룹의 주소를 표기(호스트가 아님)



**IGMP 동작 과정**

자신이 IGMP 메시지에 표시된 멀티캐스트 주소의 멤버임을 다른 호스트와 라우터에 알리기 위해

IGMP를 사용한다.



만약 가입을 원하면 해당 멀티캐스트 주소를 표기한 IGMP 보고 메시지를 전송하고,

IGMP 헤더에 Group Address에 가입을 원하는 멀티캐스트 주소를 기록한다.



**-그룹 호스트 관리**

특정 그룹에 속한 호스트 목록을 관리하기 위해서 IGMP 질의 메시지를 사용해 주기적으로 확인해야 한다.

호스트는 본인이 그룹에 속해있다는 의미로 질의 메시지에 대한 응답 메시지를 보내야 하는데

응답이 이루어지지 않으면 그룹에서 탈퇴한 것으로 간주한다



**-IGMP 메시지 전송**

IP 프로토콜과 동등한 계층의 기능을 수행하지만 ICMP와 마찬가지로 바로 전달되는 것이 아닌

IP 패킷에 캡슐화되어 보내진다.
