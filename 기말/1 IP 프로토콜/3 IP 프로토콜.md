IP는 호스트의 주소 표기, 패킷 분할에 관한 기능을 지원하지만, 단대단 형식의 오류 제어나

흐름 제어 기능은 제공하지 않는다.



IP 프로토콜에서 라우터 간의 패킷을 중개할 때는 Best Effort라는 원칙에 따라 전송하는데,

이 방ㅅㅣㄱ은 전송 패킷이 수신 호스트에 100% 도착하는 것을 보장하지 않는다.

따라서 IP 프로토콜에서 제공하지 않는 전송 오류 문제를 상위 계층에서 고려해야 한다.



IP 프로토콜의 특징

비연결형 서비스를 제공한다.

패킷을 분할/병합하는 기능을 수행하기도 한다.

데이터 체크섬은 제공하지 않고, 헤더 체크섬만 제공한다.

Best Effort 원칙에 따른 전송 기능을 제공한다.



**IP 헤더 구조**



**-DS(Differentiated Service)/ECN(Explicit Congestion Notification)**

DS와 ECN이 도입되기 전에는 8비트의 Service Type 필드로 정의되어 우선순위, 지연, 전송률, 신뢰성 등의

값을 지정할 수 있었다. Service Type 필드는 IP 프로토콜이 사용자에게 제공하는 서비스의 품질에 관련된 내용을

표현한다.



0\~2/3/4/5/6\~7 총 8비트로 구성되어 있다.

0\~2는 우선순위를 나타낸다.(111: 가장 높음)

3은 지연율을 나타낸다.(0은 보통의 지연, 1은 낮은 지연)

4는 전송률을 나타낸다.(0은 보통의 전송률, 1은 높은 전송률)

5는 신뢰성을 나타낸다.(보통의 신뢰성,  1은 높은 신뢰성)

6\~7은 예약 값이다.



**차등 서비스 개념이 도입**되면서 Service Type 필드는 6비트의 **DS** 필드와 2비트의 **ECN** 필드로 **새로 정의**되었다.

이는 인터넷에서 다양한 트래픽 요구 조건을 필요로 하는 서비스들에 대해 서로 다른 수준의 QoS를 지원하기 위함이다.

(Quality of Service)



DS를 쓰기 위해 서비스 제공자와 서비스 이용자 사이에 서비스 등급에 대한 합의가 이루어지며,

동일한 DS 값을 갖는 트래픽들은 동일한 서비스 등급으로 처리된다.

이렇한 처리 과정은 라우터에 의해 이루어지고 기존 응용 프로그램들을 변경할 필요가 없다.



**-ECN 필드 값의 의미**

00 : IP 패킷이 ECN 기능을 사용하지 않음

01(ECT 1) : TCP 프로토콜도 ECN 기능을 지원한다는 의미

10(ECT 0) : TCP 프로토콜도 ECN 기능을 지원한다는 의미

11(CE: Congestion Experienced) : 라우터가 송신 호스트에 혼잡을 통지할 때 사용



**-패킷 분할**

패킷 분할과 관련된 필드는 Identification, DF, MF, Fragment Offset이 있다.



**-Identification**

분할한 패킷들에 동일한 고유 번호를 부여해서 수신 호스트가 Identification 번호가 같은

패킷을 다시 병합할 수 있도록 해준다.

패킷을 분할하지 않으면 값을 하나씩 올리면서 분할되지 않은 패킷들을 구별하면서 전송한다.



**-DF(Don't Fragment)**

패킷이 분할되지 않도록 한다.

값이 1이면 패킷 분할을 막을 수 있다. 수신 호스트가 분할된 패킷들을 병합하지 못하는 경우 사용한다.

따라서 중간 경유 네트워크에서는 자신이 처리 가능한 크기보다 큰 패킷에 DF필드가 1인 경우

분할 기능을 수행하지 않고 패킷을 버린다.



**-MF(More Fragment)**

분할된 패킷들을 전송할 때 도착한 패킷이 마지막임을 알리는 필드이다.

1인 경우 아직 분할된 패킷들이 남아있다는 의미이고 0인 경우 지금 도착한 패킷이

분할된 마지막 패킷임을 알린다.



\-**Fragment Offset**

분할된 패킷의 내용이 원래 분할 전 데이터에서 위치하는 상대 주소 값을 의미한다.

값이 8바이트의 배수이므로,  값이 64라면 원래 데이터에서 64\*8 = 512 번째에 위치한다.



**주소 관련 필드**

Source Address와 Destination Address가 있다.

IP 주소 체계는 크게 5가지 종류가 있으며 A,B,C,D,E 클로스로 나뉜다

A,B,C는 유니 캐스팅에서 사용하고, 클래스 D는 멀티캐스팅에서 이용한다.

클래스 E는 향후의 새로운 응용 환경을 위해 잠정적으로 예약된 클래스이다.

클래스는 클래스 구분자, network, host로 구성되어 있다.

**-network**

네트워크 주소이다. 전세계적으로 유일한 번호가 모든 컴퓨터 네트워크에 할당된다.



**-host**

네트워크 주소가 결정되면 하위의 호스트 주소를 의미하는 host 비트 값을 개별 네트워크의 관리자가 할당한다.

클래스 A는 host 비트의 크기가 크기 때문에 규모가 큰 네트워크에서 사용하고 C는 작은 네트워크에서 사용한다.



**기타 필드**

Version Number: IP 프로토콜의 버전 번호이다. IPv4는 버전 4이다.

Header Length: IP 프로토콜의 헤더 길이를 32비트 워드 단위로 표시한다.

헤더의 최소 길이는 5이다.

Packet Length: IP 헤더를 포함해 패킷의 전체 길이를 나타낸다.

Time To Live: 패킷이 전달되는 동안의 수명을 의미한다.

Transport, Header Checksum, Options, Padding



**패킷 분할**

하부 계층의 관점에서 보면, TCP에서 설정하는 논리적인 가상 연겨ㄹ은 여러 종류의 네트워크를 거쳐

설정된다. 그러나 TCP 프로토콜은 패킷 전송 과정에 위치하는 네트워크 유형에 따라 패킷 크기를 조절하기가

쉽지 않으므로 IP 프로토콜에서 이 기능을 수행해야 한다.

하부 계층인 데이터 링크 계층 프로토콜의 프레임은 크기가 프로토콜마다 다르다.

따라서 상위 계층에서 내려온 데이터를 계층2의 프레임 틀에 담을 수 있도록

IP 프로토콜에서 분할 과정을 거친 후에 전송하고, 수신 측에서 다시 병합해야 한다.



분할 기능은 전송 경로에 위치한 라우터에 의해 수행된다.

라우터의 좌우에 연결된 LAN이 서로 다를 수 있기 때문에 데이터 링크 계층에 위치한 프레임 크기가

프로토콜마다 달라진다. 따라서 다른 LAN을 사용하는 라우터에게는 그에 맞는 크기로 프레임을 분할해야 한다.

IP 헤더를 제외한 전송 데이터의 크기가 380바이트이고, 패킷은 최대 크기가 128바이트라고 가정

Identification은 1254

구분자, Packet Length, MF, Fragment Offset

offset 계산

왜인지를 모르겠으나 IP 헤더가 20바이트란다.

최대 크기(128) - IP 헤더 = 108바이트 

하지만 108은 8로 나누어 떨어지지 않는다.(최소 크기가 8임)

그래서 가장 가까운 104가 분할된 패킷이 담을 수 있는 최대 데이터 크기이다.

104/8 = 13이므로 offset은 13이다.



마지막 패킷의 크기 계산

전송 데이터의 크기가 380바이트인데 분할된 패킷에 담긴 순수 데이터는 104 \* 3= 312 바이트이다.

따라서 헤더를 제외한 전송 데이터는 68 바이트이다.

여기에 분할된 패킷에 각 헤더가 따로 붙기에 68+20 = 88바이트가 총 패킷의 길이이다.
