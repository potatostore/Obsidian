파일을 전체로 다운로드하는 것이 아닌 실시간으로 다운로드하는 시대가 되면서

데이터그램 변형이나 분실 오류를 복구하는 기능이 상대적으로 덜 중요해졌다.



**도착 순서, 패킷의 지연 간격(지터) 분포의 균일성, 데이터 압축에 의한 전송 정보량의 최소화가 중요하다.**



기존의 TCP와 UDP는 위의 조건을 만족하기에 부적절하기에 새로운 프로토콜이 필요하게 되었다.



**RTP(Real Time Protocol)은** 실시간 멀티미디어 데이터의 전송을 지원한다.

RTP는 유니캐스팅뿐 아니라 멀티캐스팅도 지원한다



**특징**

1. 불규칙하게 수신되는 데이터의 순서를 정렬하기 위해 타임 스탬프 방식을 사용
2. 프로토콜의 동작이 응용 프로그램의 라이브러리 형태로 구현되는 ALF(Application Level Framing)을 사용한다.

하지만 자원 예약이나 QoS(Quality of Service) 보장과 같은 기능은 제공하지 못한다.



실시간 서비스는 전송 시간이 중요하기 때문에 송신 프로세스가 전송한 데이터의 전송 간격이 수신 프로세스에

그대로 유지되도록 하는 것이 중요하다.



**-버퍼의 역할**

송신 프로세스가 전송한 데이터는 데이터그램 사이의 시간 간격이 일정하다가 인터넷 네트워크를 거쳐

수신 프로세스에 전달되는 과정에서 간격이 불규칙해진다.

따라서 **수신 측**에서는 간격이 가변적인 데이터를 수신 프로세스에 즉시 전송하지 않고,

지연 버퍼를 사용해 데이터의 시간 간격을 일정하게 보정한다. 이때 도착한 데이터의 시간이

실시간 재생에서 요구하는 일정 범위보다 늦으면 해당 데이터를 버린다.

따라서 데이터 전송을 완료하기 위해서는 전송 대역폭을 충분히 확보해야 한다.

이런 문제를 송수신 프로세스 사이의 **지연 시간**에 관한 문제라고 한다.



**-지터**

송신 프로세스는 일정하게 간격으로 데이터를 전송한다.

하지만 수신 프로세스는 일정한 간격으로 데이터를 받지 않는다.

그렇기에 수신 프로세스가 버퍼에 도착한 데이터를 쌓아두고 일정한 간격으로 처리를 하는 식으로

실제로 체감되는 지연 시간을 일정하게 유지한다.

이때 **지터**는 데이터그램 간격의 분포를 의미한다.



**RTP의 데이터 전송**

기본적으로 RTP는 UDP 위에서 구현되기에 UDP가 지원하지 않는 도착 순서 변경이나 전송 오류는

RTP에서 처리하도록 되어 있다.

이때 전송하는 데이터의 타입에 따라서 RTP가 ALF로 각각 구현되고 각 모듈들이 모여 RTP가 된다.

RTP는 다수의 사용자가 **하나의 세션에 참여**해 **서로** 실시간 **데이터를 전송**하는 것을 지원한다.

**세션은 그룹을 형성한 멀티캐스트 전송을 지원하기 위해 사용한다.**



**-RTP 릴레이**

**믹서와 트랜슬레이터**라는 두 종류의 릴레이를 지원한다.

릴레이란 데이터 전송 과정에서 송수신 프로세스가 데이터를 직접 전송할 수 없는 상황이 발생했을 때

데이터를 중개하는 기능이다.

방화벽이 설치되거나 데이터 형식이 다른 경우 릴레이가 필요한 처리를 수행해 데이터 전송을 지원한다.



**-믹서**

믹서는 여러 송신 프로세스로부터 RTP 데이터그램 스트림을 받아 이들을 적절히 조합해

새로운 데이터그램 스트림을 생성한다.

이때, 데이터 형식이 변하거나 믹싱 기능이 수행될 수 있다.

RTP는 타임스탬프를 사용하기에 새로운 데이터그램 스트림에도 시간 정보를 제공한다.

또한 믹서 자신이 해당 데이터그램의 시간 정보를 제공했음을 명시해야 한다.



**-트랜슬레이터**

트랜슬레이터는 입력된 각 RTP 데이터그램을 하나 이상의 출력용 RTP 데이터그램으로 만들어주는 장치이다.

이 과정에서도 데이터 형식이 변할 수 있다.

예를 들어 특정 수신 프로세스가 고해상도 비디오 신호를 처리할 능력이 없으면, 트랜슬레이터가 이를 저해상도

신호로 변환해 수신 프로세스가 처리할 수 있도록 도와준다.



**-믹서와 트랜슬레이터의 차이**

스트림은 데이터그램을 연속으로 전송한다는 의미인데, 실시간 서비스에서 사용하는

연속 데이터그램은 시간에 민감하다. 이때 믹서는 데이터그램의 믹싱에 관한 문제를 다루기에

타임스템프 정보를 명시하지만 트랜슬레이터의 경우 스트림은 건들지 않고 특정한 변환 작업을 수행한다.



## **RTP 헤더 구조**



**실시간 서비스**를 위한 RTP는 유니캐스트뿐 아니라 **멀티캐스트 전송**도 지원한다.

따라서 데이터 형식에는 멀티캐스트 그룹에서 누가 데이터를 전송했는지 확인하는

**Source Identifier(송신 구분자)**필드가 존재한다.

또한 수신 프로세스에서 지연 버퍼를 사용해 타이밍 관계를 조절할 수 있도록 **TimeStamp** 필드도 지원한다.



총 12바이트로 구성되며 Version,Padding,Extension,CSRC Count, Marker, Payload Type, Sequence Number

Timestamp, SSRC(Synchronization Source) Identifier, CSRC(Contributing Source) Identifier이 있다.



Timestamp는 RTP 페이로드에 포함된 데이터의 생성 시기를 나타낸다.

시간 단위는 페이로드의 종류에 영향을 받으며, 송싱 프로세스에서 사용하는 클록에 의해 발생한다.



SSRC Identifier는 임의의 세션 내에서 RTP 페이로드의 발신지가 어디인지를 구분하는 번호로

랜덤하게 생성되는 32비트 숫자이다.



CSRC Identifire는 믹서가 제공하는 지원자이다. 여기에는 여러 데이터를 조합해 만든 새로운

데이터 스트림이 어떤 데이터 타입인지에 대한 인코딩 종류가 저장된다.

(audio, video, JPEG video등이 데이터 타입의 예시이다)

데이터 타입이 다르면 별도의 RTP 세션을 사용해서 데이터를 전송한다.



## **RTP 제어 프로토콜**

RTP 제어 프로토콜은 RTP 데이터 전송 프로토콜과 구분하기 위해 RTCP(RTP Control Protocol)이라 부른다.

데이터 전송 프로토콜은 세션 참가자 사이의 멀티캐스트 기능을 이용한 사용자 데이터의 전송을 담당하고

RTCP는 제어와 관련된 기능을 수행한다.



RTCP는 RTP와 같이 UDP를 하부 전송 계층으로 사용한다.



**RTCP의 기능**



**-QoS와 혼잡 제어**

서비스 품질을 위해 데이터 분배 과정에서 발생하는 서비스 품질에 대한 피드백을 지원한다.

멀티캐스팅 과정에서 세션 멤버의 데이터 송수신 과정이 어떻게 이루어졌는지를 판단하는데

이때 송수신 프로세스가 관련 보고서를 작성한다. 전송률, 패킷 분실이나 지터 등의 정보가 포함된다.



**-Identifier**

RTCP 패킷에는 송신 프로세스에 관한 구분자 정보가 포함되며, 서로 다른 세션에서 발신된 스트림 정보들을

서로 연관시키는데 사용한다.



**-세션 크기**

RTCP가 동작하기 위해서는 세션 참가자들 사이에 RTCP 패킷이 주소적으로 전송되어야 한ㄷㅏ.

이때 세션 참가자수가 증가하면 RTCP 패킷의 전송률은 감소하는데 이는 RTCP 패킷의 트래픽이 증가함에 따라

세션 전체의 트래픽이 증가하기 때문이다. 따라서 이를 제한하는데, 일반적으로 전체 세션 트래핏의 5% 이내로

유지되도록 알고리즘이 동작한다.



\-**RTCP 패킷의 종류와 역할**

Sender Report, Receiver Report - QoS를 위해 사용

Source Description - 송신 프로세스가 자신에 대한 정보를 더 많이 제공하는 용도

Goodbye - 송신 프로세스가 더 이상 존재하지 않음을 의미

Application Packet - 응용 환경에 따른 기능을 점검하기 위해 제공
